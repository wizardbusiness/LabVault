Writing RIICH documentation: 

Setup
- goal - why are you doing what you're documenting
- requirements - What do you need to have prepared in order to extract value from this documentation
- resources - anything you're looking at while you're doing the thing for instruction

Stage 1: Getting key information down
- You need to make this stage as easy on yourself as possible. 
	- As little formatting as you can get away with, time thinking about structure is frustrating. 
- Question everything
	- Anytime *you* have a question you don't know the answer to, write it down with a `>?` marker

Stage 2: First Pass - Organization and Hierarchy
- Keep moving. Don't get super hung up on any one section for now. Your goal is to nail down the structure and flow of the document, the content does not need to be finished yet. 
- Birds-eye view. Orient the reader with an overview of the tool / process being documented, the logic it adheres to and what they can expect to learn how to do. You should **also** provide an escape hatch for more experienced users / those who want to accomplish a specific goal and don't want or need to start at the beginning. Good documentation should not force the reader to follow *your* preferred flow, but should support them no matter their skill level  or objectives (within reason). 
  - *ex: Sandwiches are fundamentally 1 or more slices of bread, usually condiments and always toppings. The order is important. The bread **always** goes on the outside. Sandwiches get their name from the fact that most sandwiches have 2 slices of bread, which 'sandwich' the other ingredients between them. To get started we'll be covering how to make a **PB&J** sandwich. This sandwich is easy to make but also a classic for a reason.* 
  - **Escape Hatch** 
	  - *Sandwich Type Reference Guide* 
	  - *Sandwich-Related words and definitions*
	  - *Figuring out why your sandwich tastes bad*
	  - *Sandwich-making best practices*
	  - See the table of contents for more

- Immediately Engage the user
	- Give them an example of something **useful** that can be accomplished with the tool
		- Its best if **useful** is understood as 'what will make the the average person using this be like 'cool' when they see this example. That's *usually* something practical that teaches them the basics of using the tool. 
	- Show them how to implement that example from start to finish
- Respect the reader's time
	- What is the *least* information that the reader needs to be *most* productive
